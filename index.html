3D Secure Strategy

Fix Critical Bugs, Automatic Exemptions & Intelligent Routing
2026 Q2 · Payments Team Status: Draft — Internal Review March 2026
Fix critical bugs that break 3DS exemptions and skip logic for ~30-40% of EU merchants, implement proper CIT-only 3DS with MIT/BIN-level skip rules, then build an automatic exemption engine and intelligent 3DS routing — turning Yuno's multi-provider 3DS architecture into a measurable conversion advantage.

Problem

  1. Bug OR-694 (P0): In routing-orc, 3DS exemptions (Low-value, Recurring, Trusted Beneficiary, Secure Corporate, MOTO) are silently ignored unless a fraud provider (Forter/Riskified) is in the route. ~30-40% of EU merchants are affected — they face 100% challenge rates on transactions that should be exempt. Industry data: 10-15% cart abandonment per unnecessary challenge.
  2. Bug in gateway-integration-api (P0): The ThreeDSecure struct on the payment path lacks StrongCustomerAuthenticationExemptions. Even if OR-694 is fixed, exemptions are silently dropped before reaching acquirers.
  3. Incomplete 3DS skip logic (P0): Only CIT (Cardholder-Initiated Transactions) should go through 3DS. Today, MIT transactions (subscriptions with usage=USED) and BINs from non-mandate regions unnecessarily go through 3DS. No AUTHORIZATION_REQUIRED auto-retry exists.
  4. No automatic exemption engine: Every competitor (Adyen, Stripe, Checkout.com, Braintree) auto-applies rule-based exemptions. Yuno does nothing without a fraud provider.
  5. No 3DS analytics dashboard: Merchants have zero visibility into frictionless vs. challenge rates.
  6. No automatic 3DS routing: Yuno has 2 standalone 3DS providers (Netcetera and Checkout.com) but no intelligent routing to select the best-performing MPI.
Strategic opportunity: No competitor offers multi-provider standalone 3DS routing. Adyen, Stripe, and Checkout.com are locked to their own single 3DS engine. By building automatic routing that selects the MPI with the best frictionless/approval rate per country/BIN/issuer — without the merchant changing anything — Yuno can offer a differentiation no single-PSP competitor can replicate.
Checkout.com blocker: Checkout.com 3DS requires a processor_channel_id per merchant, which today is a manual onboarding process. Checkout.com is building an API to automate this but there is no ETA. This makes Netcetera the de facto sole MPI today.

3DS Skip Logic: Only CIT Goes to 3DS

Three independent rules evaluate whether to skip 3DS. If any rule matches, 3DS is skipped:

RuleConditionActionRationale
1Fraud provider returns OUT_OF_SCOPESkip 3DSFraud intelligence says out of scope
2Card BIN from non-mandate country (US, LATAM, etc.)Skip 3DSNo regulatory requirement
3MIT: storedCredentialTransaction.usage == USEDSkip 3DSCardholder not present to authenticate

Only CIT (usage == FIRST) goes through 3DS authentication.

Override: AUTHORIZATION_REQUIRED Auto-Retry

If a provider declines with AUTHORIZATION_REQUIRED (Visa 65, MC 1A), the issuer is explicitly demanding SCA:

  • All skip rules are overridden — the transaction MUST go through 3DS
  • If merchant has auto_retry_authorization_required enabled (default in Managed mode): Yuno automatically retries with 3DS — transparent to the merchant
  • If merchant has disabled auto-retry (Custom mode option): decline returned to merchant, who handles retry themselves
  • Max 1 automatic retry to prevent loops

Decision Flow

Payment request arrives | +-- Was this a retry after AUTHORIZATION_REQUIRED? | | | YES -> Go to 3DS unconditionally (override) | | | NO --> Evaluate skip rules: | | | +-- Rule 1: Fraud OUT_OF_SCOPE? -> Skip 3DS | +-- Rule 2: BIN non-mandate? -> Skip 3DS | +-- Rule 3: MIT (usage=USED)? -> Skip 3DS | | | +-- No rules matched -> CIT -> Go to 3DS | | | +-- Select MPI (auto-routing) | +-- Apply exemptions | +-- Authenticate | +-- Payment submitted to provider | +-- Approved -> Done +-- Declined: AUTHORIZATION_REQUIRED | | | +-- auto_retry enabled? | | YES -> Force 3DS, retry payment | | NO -> Return decline to merchant | | +-- Declined: other -> Normal decline handling

FAQ

Why is fixing the skip logic a P0 alongside OR-694?

MIT transactions (subscriptions, recurring charges) should never go through 3DS — the cardholder isn't present. Sending them to 3DS causes authentication failures. This is a basic correctness issue every competitor handles correctly.

What happens if a skipped transaction needs SCA after all?

The provider declines with AUTHORIZATION_REQUIRED. If the merchant has auto-retry enabled (default), Yuno forces 3DS and retries automatically. If disabled, the decline is returned to the merchant. This is the safety net that makes aggressive skipping safe.

Why can't merchants just add a fraud provider?

Fraud providers (Forter/Riskified) cost $50K+/year. SMB merchants shouldn't need one to apply a €30 low-value exemption — that's arithmetic, not risk scoring.

How many standalone 3DS providers does Yuno have?

Two — Netcetera (~5.8M/month) and Checkout.com (~416/month). Others handle 3DS natively within their payment flow and are not part of the standalone product.

What's the competitive moat?

Automatic multi-provider 3DS routing. No competitor can do this — they each use their own single 3DS engine. Combined with proper skip logic and automatic exemptions, Yuno offers a best-in-class 3DS product.
